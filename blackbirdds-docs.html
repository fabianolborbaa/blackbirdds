Foundation Tokens

Design tokens are the single source of truth for BlackBirdDS visual decisions — color, typography, spacing, shape, elevation, and layout. Every value in this document maps directly to a Figma variable or style.

What are Design Tokens?
  • Tokens are named values — instead of hardcoding #155DFC, you reference color/accent/default.
  • They are platform-agnostic — the same token works in Figma (as a variable), CSS (as a custom property), and code (as a constant).
  • Semantic tokens reference primitives — color/accent/default resolves to colors/brand/600, which is #155DFC.
  • Light and Dark mode — Semantic tokens have two values. Swap the mode and the entire UI updates automatically.
Color · Primitives

Primitive Colors

The raw color palette. Each family runs from 050 (lightest) to 950 (darkest) with 11 steps. Primitives are the foundation — never use them directly in components; reference semantic tokens instead.

— color tokens 13 color families 11 steps per family
✓ Do

Reference semantic tokens in components: color/accent/default, color/text/primary.

✗ Don't

Don't use primitive tokens in component fills: avoid colors/brand/600 directly in UI code.

Color · Semantic

Semantic Colors

Semantic tokens carry intent and adapt to Light and Dark mode. Use these tokens in all components and layouts — they are the contract between design and code.

33 semantic tokens 2 modes: Light + Dark 9 categories
How to use semantic tokens
  • Background tokens set the canvas color of pages and sections.
  • Surface tokens are for cards, panels, and elevated containers.
  • Text tokens define hierarchy — use primary for body, secondary for labels, tertiary for hints.
  • Border tokens separate content areas and indicate interactive focus.
  • Accent tokens drive the brand presence — primary buttons, links, active states.
  • Semantic feedback tokens (Success/Warning/Error/Info) are for alerts, badges, and validation states.
Typography · Scale

Type Scale

13 sizes from text-xs (12px) to text-9xl (128px), all using IBM Plex Sans JP with a 1.5× line height. The scale is designed for clear visual hierarchy across product interfaces.

13 size steps 7 weights per size 91 total styles IBM Plex Sans
Usage guidelines
  • xs / sm (12–14px) — captions, labels, badge text, helper text.
  • base / lg (16–18px) — body copy, the default reading size.
  • xl / 2xl (20–24px) — section headings, card titles.
  • 3xl – 5xl (30–48px) — page headings, hero copy.
  • 6xl+ (60px+) — display type, marketing, large numerals.
Typography · Weights

Font Weights

IBM Plex Sans supports 6 weights from Thin (100) to Bold (700). Use weight to establish visual hierarchy and signal emphasis — without relying on size alone.

6 available weights IBM Plex Sans
Weight usage
  • Thin (100) / Light (300) — display headlines, decorative large text.
  • Regular (400) — all body copy; the default weight.
  • Medium (500) — UI labels, nav items, form labels.
  • SemiBold (600) — subheadings, badge text, button labels.
  • Bold (700) — page titles, emphasized values, active states.

Spacing · Scale

Spacing Scale

All spacing values are multiples of 4px (1 base unit). The scale follows a Tailwind-compatible naming convention and covers the full range from 1px to 384px.

31 spacing tokens Base unit: 4px Range: 1px → 384px
Spacing rules
  • Micro (1–6px) — icon gaps, indicator dots, border offsets.
  • Small (8–14px) — internal padding within compact components (badges, chips).
  • Medium (16–32px) — component padding, list item gaps, button padding.
  • Large (36–64px) — section gaps, card padding, group spacing.
  • XL+ (80–384px) — page margins, hero padding, layout columns.
Shape · Border Radius

Border Radius

Nine radius steps from sharp (0) to pill (9999px). Consistent corner rounding creates visual harmony across all components.

9 radius tokens Range: 0 → 9999px
Radius guidelines
  • none (0) — data tables, full-bleed containers, system chrome.
  • sm (2px) — small tags, inline code, tooltips.
  • rounded (4px) — form inputs on tight layouts.
  • md (6px) — default for most inputs and compact cards.
  • lg (8px) — standard card radius, dropdown menus.
  • xl (12px) — modal dialogs, large cards.
  • 2xl / 3xl — feature cards, hero images.
  • full (9999px) — badges, pills, avatar circles.
Elevation · Shadows

Shadows

12 shadow tokens define elevation and focus. The neutral scale (xs → 2xl) conveys depth; colored shadows are used for brand emphasis and interactive feedback.

12 shadow tokens 6 neutral elevations 6 colored shadows
Elevation system
  • shadow-xs — subtle depth for borders that need a little lift (cards on white).
  • shadow — default card elevation.
  • shadow-md — dropdowns, tooltips, popovers.
  • shadow-lg — modals, drawers, side panels.
  • shadow-xl / 2xl — fullscreen overlays, command palettes.
  • Colored shadows — used on primary buttons and interactive states to reinforce brand color.
Layout · Grid System

Grid System

9 grid presets cover the full range of layout needs — from dense 12-column editorial layouts to simple single-column mobile stacks. All grids use a gutter-based system without fixed margins.

9 grid configurations Column-based system
Grid guidelines
  • 12-column grids are the most flexible — they support 2, 3, 4, 6 and 12 column layouts.
  • gap-16 (64px gutter) is for spacious editorial layouts; gap-4 (16px) for dense dashboards.
  • 8 and 6 columns work well for content-focused pages with a sidebar.
  • 1-column is the default for mobile breakpoints.

✓ Do

Use Primary for the single most important action on a screen — submit, save, confirm.

✗ Don't

Don't place two Primary buttons side by side. Only one action should be primary at a time.

✓ Do

Use Destructive only for irreversible actions (delete, remove). Always confirm before executing.

✗ Don't

Don't use Destructive for cancel or undo — those are recoverable actions and should use Ghost or Secondary.

✓ Do

Use Ghost for low-priority or tertiary actions in toolbars, table rows, and dense UI.

✗ Don't

Don't use Ghost as the only button in a form — it won't attract enough attention for the primary action.

Accessibility notes
  • All buttons must have a visible focus ring for keyboard navigation.
  • Disabled buttons should still be visible (not hidden) — use opacity, not display:none.
  • Icon-only buttons must include an aria-label describing the action.
  • Minimum touch target: 44×44px — use padding to extend if necessary.

Use semantic variants consistently: Success = operational, Warning = degraded, Error = outage/failure, Info = in-progress.

Don't use color alone to convey meaning — always include a text label so the badge is understandable without color vision.

✓ Do

Always include a visible label above the field. Never rely on placeholder text as a substitute for a label.

✗ Don't

Don't use placeholder text as a label — it disappears when the user starts typing, leaving them without context.

✓ Do

Show validation errors inline with specific, actionable messages: "Enter a valid email" not "Invalid input".

✗ Don't

Don't show error states before the user has had a chance to interact with the field (avoid eager validation).

Accessibility notes
  • Every input must have a programmatically associated label via for/id or aria-label.
  • Error messages must be linked to the field via aria-describedby.
  • Use aria-invalid="true" on the input element when in error state.
  • Error states must never rely on color alone — pair the red border with the error message text.
  • Disabled inputs should have aria-disabled="true" if using a non-native element.
